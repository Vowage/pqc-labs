Concept · Why Hybrid? · The Both-or-Zero Guarantee

One Algorithm Is Never Enough During a Transition

The intuition in 5 sentences. Then a comparison table. Then a micro-check.

💡 The Core Idea — Both-or-Zero A hybrid key exchange runs two independent key exchange algorithms and combines their outputs. The final session key is secure as long as at least one component algorithm remains secure. If quantum computers break ML-KEM tomorrow, the X25519 component still protects you against classical adversaries. If a classical vulnerability is found in X25519, ML-KEM still protects you against quantum adversaries. An attacker must break both simultaneously to compromise the session.
🏛️
Classical Only
X25519 alone. Fast, small. Secure today. A quantum computer with Shor's algorithm breaks it completely. No fallback.
✗ Quantum-vulnerable
⚛️
PQ Only
ML-KEM alone. Quantum-safe. But lattice cryptography is young — 30 years less classical scrutiny than Diffie-Hellman. If a classical break is found, no fallback.
⚠ Unknown classical risk
🔀
Hybrid
X25519 + ML-KEM-768 combined via HKDF. Must break both simultaneously. The stronger component always wins.
✓ Best-of-both security
⏳ The Migration Reality Google Chrome deployed X25519+ML-KEM-768 as the default TLS key exchange in August 2024. Cloudflare enabled it across their network simultaneously. They did not wait for complete PQ migration — they deployed hybrid, getting quantum protection immediately while maintaining full classical compatibility with every existing server and client. This is the recommended transition path from NIST and IETF.

✅ The Dominant Strategy Hybrid is the cryptographic dominant strategy during the transition period: it is never worse than the stronger component and is strictly better whenever one component is compromised. The overhead cost (+2304 bytes in a TLS handshake) is justified by the security gain under any threat model.

Two shared secrets must be combined into one session key. The combination method is not trivial — doing it wrong can reduce the hybrid to the security of just one component.

hkdf_combination.py
from cryptography.hazmat.primitives.kdf.hkdf import HKDF
from cryptography.hazmat.primitives import hashes
# Two independent shared secrets from two algorithms
k_x25519 = bytes(32) # from X25519 key exchange
k_mlkem = bytes(32) # from ML-KEM-768 decapsulation
# HKDF combines them cryptographically — not just concatenation
hkdf = HKDF(
    algorithm=hashes.SHA256(),
    length=32,
    salt=b"hybrid-pqc-v1",
    info=b"X25519+ML-KEM-768 TLS1.3 hybrid",
)
hybrid_key = hkdf.derive(k_x25519 + k_mlkem)
# Result: 32-byte key whose security = max(k_x25519, k_mlkem)
✗ Wrong: naive concatenation key = k_x25519 + k_mlkem

This produces a 64-byte key. But the security of the first 32 bytes is completely independent of the last 32 bytes. If an attacker learns k_x25519, they can derive the first half. The "combined" key provides no cryptographic binding between the two components.
✓ Correct: HKDF key = HKDF(k_x25519 || k_mlkem)

HKDF mixes both inputs together cryptographically. The output is uniformly random and depends on both inputs simultaneously. An attacker who knows k_x25519 still learns nothing about the output without also knowing k_mlkem.

Hybrid key exchange is already deployed at production scale. This is not a theoretical proposal.

🌐
Google Chrome + Cloudflare — August 2024
Chrome shipped X25519MLKEM768 as the default TLS key exchange. Cloudflare enabled support simultaneously. The hybrid key share adds approximately 1200 bytes to the TLS ClientHello — imperceptible on modern connections.
🔐
OpenSSH 9.0+ — 2022
OpenSSH added sntrup761x25519-sha512 as a hybrid option, combining a NTRU-based PQC algorithm with X25519. This predates ML-KEM standardization but follows the same hybrid pattern.
📋
IETF RFC (in progress) — TLS 1.3 Hybrid Key Exchange
IETF is standardizing the hybrid key exchange format for TLS 1.3 via the hybrid-kem draft. The pattern you implement in this lab is the reference design used in that specification.

Exercise · Hybrid X25519 + ML-KEM-768 Implementation

Build the Full Hybrid Key Exchange

Run each phase of the hybrid exchange. Both X25519 and ML-KEM-768 run in parallel. HKDF merges the two secrets. Confirm both parties derive the same hybrid key.

lab4_hybrid.py — Full Implementation
import oqs
from cryptography.hazmat.primitives.asymmetric.x25519 import X25519PrivateKey
from cryptography.hazmat.primitives.kdf.hkdf import HKDF
from cryptography.hazmat.primitives import hashes, serialization
# Phase 1 — Alice generates both keypairs
alice_x25519_sk = X25519PrivateKey.generate()
alice_x25519_pk = alice_x25519_sk.public_key()
alice_kem = oqs.KeyEncapsulation('ML-KEM-768')
alice_kem_pk = alice_kem.generate_keypair()
# Phase 2 — Bob performs X25519 + ML-KEM encapsulation
bob_x25519_sk = X25519PrivateKey.generate()
k_x25519_bob = bob_x25519_sk.exchange(alice_x25519_pk)
ct_mlkem, k_mlkem_bob = alice_kem.encap_secret(alice_kem_pk)
# Phase 3 — Alice recovers both secrets
k_x25519_alice = alice_x25519_sk.exchange(bob_x25519_sk.public_key())
k_mlkem_alice = alice_kem.decap_secret(ct_mlkem)
# Phase 4 — Both combine via HKDF
hybrid_key_alice = HKDF(SHA256, 32, salt, info).derive(k_x25519_alice + k_mlkem_alice)
hybrid_key_bob = HKDF(SHA256, 32, salt, info).derive(k_x25519_bob + k_mlkem_bob)
assert hybrid_key_alice == hybrid_key_bob # ← this is what you'll verify

Hybrid Design: The Strategic Picture

Three questions that test whether you understand not just how hybrid works, but why it was designed this way and when to use it.

1. The bank in this lab's scenario has 4.2 million RSA hardware tokens that cannot be replaced until 2028. A colleague argues: "Since RSA tokens will still be in use, there's no point deploying hybrid now — we'll just migrate everything at once in 2028." What's wrong with this reasoning?
A
The colleague is correct — deploying hybrid piecemeal creates incompatibility issues that make the 2028 migration harder.
B
Harvest-Now-Decrypt-Later attacks are happening today. Every TLS session using only RSA or ECDH is being recorded by adversaries who will decrypt it retroactively once a quantum computer arrives. Deploying hybrid immediately stops future sessions from being harvested in decryptable form — even though legacy tokens still exist. The 2028 timeline may be optimistic; the quantum threat window could open before then.
C
The colleague's plan is valid but requires filing an exception with NIST to delay PQC deployment beyond the 2030 recommendation.
D
Hardware tokens are irrelevant to TLS session security — the two systems are independent.
2. After deploying hybrid X25519+ML-KEM-768, a security researcher publishes a paper showing ML-KEM-768 has a theoretical weakness that reduces its effective security from 192 bits to 100 bits — but it is not yet practically broken. What action should the bank take?
A
Immediately roll back all systems to classical-only X25519 until a patched version of ML-KEM is released.
B
Continue with hybrid unchanged — 100-bit security is still well above practical attack levels and the hybrid X25519 component provides full classical protection regardless.
C
Monitor the situation and begin planning an upgrade to ML-KEM-1024, but do not roll back — the hybrid system remains more secure than classical-only because X25519 provides full classical protection and the weakened ML-KEM-768 still provides significant quantum resistance. The hybrid design means the bank is not fully exposed even during the upgrade planning period. This is exactly the grace period that hybrid mode was designed to provide.
D
Switch entirely to ML-KEM-1024 only and remove X25519 to eliminate the weakened component from the hybrid.
3. Why does the salt parameter in HKDF matter for the hybrid key combination? Could the same salt value be reused across multiple sessions?
A
The salt is cosmetic — it labels the key material for debugging purposes and has no cryptographic significance.
B
The salt must be unique per session; reusing it is a critical vulnerability that immediately breaks the hybrid key.
C
The salt in HKDF serves as a domain separator and entropy whitener — it ensures the derived key is uniform even if the input key material is biased. For hybrid key exchange, the salt is typically a fixed protocol label (like "X25519+ML-KEM-768 hybrid v1"), not a per-session random value. This is because the per-session randomness already comes from the X25519 and ML-KEM key material. Reusing the salt is fine and by design — it is the input key material (k_x25519 and k_mlkem) that must be unique per session.
D
The salt is concatenated with the output, extending the effective key length to 64 bytes.
