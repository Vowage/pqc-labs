10–15 Minute Concept Overview

How Kyber Works — Without the Math

You don't need to prove theorems to deploy cryptography. You need the right mental models. Here's what Kyber actually does and why it's quantum-resistant.

Key Encapsulation Mechanism (KEM)

Before two parties can communicate securely, they need a shared secret key. RSA solved this by encrypting a key directly. A KEM solves it differently — and more cleanly.

Step 1: KeyGen
🔑 Alice
Generates public key + secret key
→
Step 2: Encapsulate
📦 Bob
Uses Alice's public key to create:
ciphertext + shared secret
→
Step 3: Decapsulate
🔓 Alice
Uses secret key on ciphertext to
recover the same shared secret
🔑 KEM vs. Key Exchange — What's the Difference? In traditional Diffie-Hellman key exchange, both parties contribute to the shared secret. In a KEM, one party (Bob) generates the shared secret and "encapsulates" it for the other (Alice). The result is the same — both parties end up with an identical shared key for symmetric encryption — but the mechanism is simpler and better suited to lattice-based constructions.

Why Not Just Encrypt the Key With RSA?

RSA key transport (encrypting a session key directly with RSA) works, but it's being replaced for two reasons. First, RSA is broken by quantum computers. Second, KEM designs provide IND-CCA2 security (resistance to chosen-ciphertext attacks) more naturally than naive RSA encryption. ML-KEM gives you both quantum resistance and better security properties.

📌 What You'll Implement Today In this lab, you'll call three functions: KeyGen() → generates the key pair, Encaps(pk) → produces ciphertext + shared secret, Decaps(sk, ct) → recovers the shared secret. That's it. Three function calls to replace RSA key exchange.

Lattice-Based Cryptography — The Intuition

Kyber's security is based on problems involving lattices — regular grid-like structures in high-dimensional space. Here's the intuition without the linear algebra.

Think of it like a city grid.

Imagine a perfect grid of streets in a 2D city. Given any coordinate, it's easy to find the nearest intersection. Now imagine that grid exists in 500+ dimensions, and instead of nice right angles, the grid lines are skewed at strange angles. Finding the nearest lattice point in this warped, high-dimensional space is extraordinarily hard — even for quantum computers.

❌ RSA's Problem (broken by quantum)

Factoring: Given N = P × Q, find P and Q. This has deep mathematical structure that Shor's algorithm exploits — specifically, the periodicity of modular exponentiation.

✓ Kyber's Problem (quantum-resistant)

Shortest Vector: Find the shortest non-zero vector in a high-dimensional lattice. No mathematical structure for quantum algorithms to exploit. Brute-force search is the only option — and it's exponential.

🌾 Remember the Haystack from Lab 1 RSA is a combination lock — quantum computers have the master key. Lattice problems are a haystack where everything looks like hay. No quantum shortcut exists. At 500+ dimensions, even quantum brute-force takes longer than the age of the universe.

Learning With Errors (LWE) — Kyber's Foundation

Kyber is specifically built on the Module Learning With Errors (MLWE) problem. Here's the intuition.

Without Noise: Easy

Imagine a system of linear equations: b = A·s. Given A (public) and b (output), finding the secret s is straightforward — it's just solving a system of equations. Any computer can do this instantly.

Clean outputs: pattern is detectable

With Noise: Impossibly Hard

Now add small random noise: b = A·s + e. The noise "e" is tiny, but it makes the system unsolvable. You can't distinguish the noisy output from random garbage — and neither can a quantum computer.

Noisy outputs: pattern destroyed

⚠ The Noise Is the Security The entire security of Kyber comes from that small amount of noise. Without it, the problem is trivially solvable. With it, no known algorithm (classical or quantum) can efficiently recover the secret. The noise must be small enough that legitimate parties can still "round off" the errors to recover the shared secret, but large enough to prevent adversaries from doing so.
🎯 Why "Learning" With Errors? The name comes from machine learning. Imagine trying to learn a hidden rule from data — but every data point has a random error mixed in. If the errors are large enough, no learning algorithm can figure out the rule. That's the fundamental hardness assumption Kyber relies on.

step1_keygen.py
import oqs import time from cryptography.hazmat.primitives.asymmetric import rsa, padding from cryptography.hazmat.primitives import hashes, serialization # ──── ML-KEM (Kyber-768) Key Generation ──── print("=== ML-KEM-768 (Kyber) Key Generation ===") kem = oqs.KeyEncapsulation("ML-KEM-768") start = time.perf_counter() public_key_kyber = kem.generate_keypair() kyber_keygen_time = (time.perf_counter() - start) * 1000 print(f"Public key size: {len(public_key_kyber)} bytes") print(f"Secret key size: {kem.export_secret_key().__len__()} bytes") print(f"KeyGen time: {kyber_keygen_time:.3f} ms") # ──── RSA-2048 Key Generation ──── print("\n=== RSA-2048 Key Generation ===") start = time.perf_counter() rsa_private = rsa.generate_private_key( public_exponent=65537, key_size=2048 ) rsa_keygen_time = (time.perf_counter() - start) * 1000 rsa_public = rsa_private.public_key() rsa_pub_bytes = rsa_public.public_bytes( serialization.Encoding.DER, serialization.PublicFormat.SubjectPublicKeyInfo ) rsa_priv_bytes = rsa_private.private_bytes( serialization.Encoding.DER, serialization.PrivateFormat.PKCS8, serialization.NoEncryption() ) print(f"Public key size: {len(rsa_pub_bytes)} bytes") print(f"Private key size: {len(rsa_priv_bytes)} bytes") print(f"KeyGen time: {rsa_keygen_time:.3f} ms")

# ──── ML-KEM Encapsulation & Decapsulation ──── print("=== ML-KEM-768 Encapsulate / Decapsulate ===") kem = oqs.KeyEncapsulation("ML-KEM-768") public_key = kem.generate_keypair() # Encapsulate: Bob creates shared secret + ciphertext start = time.perf_counter() ciphertext, shared_secret_enc = kem.encap_secret(public_key) kyber_encaps_time = (time.perf_counter() - start) * 1000 print(f"Ciphertext size: {len(ciphertext)} bytes") print(f"Shared secret size: {len(shared_secret_enc)} bytes") print(f"Encaps time: {kyber_encaps_time:.3f} ms") # Decapsulate: Alice recovers shared secret start = time.perf_counter() shared_secret_dec = kem.decap_secret(ciphertext) kyber_decaps_time = (time.perf_counter() - start) * 1000 print(f"Decaps time: {kyber_decaps_time:.3f} ms") print(f"Secrets match: {shared_secret_enc == shared_secret_dec}") print(f"Shared secret (hex): {shared_secret_enc.hex()[:64]}...") # ──── RSA Key Transport (for comparison) ──── print("\n=== RSA-2048 Key Transport ===") import os session_key = os.urandom(32) # 256-bit session key start = time.perf_counter() rsa_ciphertext = rsa_public.encrypt( session_key, padding.OAEP( mgf=padding.MGF1(algorithm=hashes.SHA256()), algorithm=hashes.SHA256(), label=None ) ) rsa_encrypt_time = (time.perf_counter() - start) * 1000 start = time.perf_counter() recovered_key = rsa_private.decrypt( rsa_ciphertext, padding.OAEP( mgf=padding.MGF1(algorithm=hashes.SHA256()), algorithm=hashes.SHA256(), label=None ) ) rsa_decrypt_time = (time.perf_counter() - start) * 1000 print(f"RSA ciphertext size: {len(rsa_ciphertext)} bytes") print(f"Encrypt time: {rsa_encrypt_time:.3f} ms") print(f"Decrypt time: {rsa_decrypt_time:.3f} ms") print(f"Keys match: {session_key == recovered_key}")

Scenario 1: Quantum Computer Breaks X25519

A cryptographically relevant quantum computer arrives and breaks X25519 (Shor's algorithm on the ECDLP). In a pure-classical system, this is catastrophic — all traffic is exposed. But in a hybrid system, the attacker also needs to break ML-KEM, which requires solving the lattice problem. The hybrid holds because the attacker must break BOTH components simultaneously.

🛡 Defense-in-Depth This is the primary reason to use hybrid during migration: even if one algorithm falls, the other protects you. The attacker faces the strongest of the two, not the weakest.
