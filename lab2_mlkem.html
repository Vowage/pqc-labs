SECTION 1 OF 6
Concept · Learning With Errors + ML-KEM Overview

Why Lattice Problems Are Quantum-Hard

The intuition in 5 sentences. Then we visualize it. Then a micro-check before you implement.

💡 The Core Idea — Learning With Errors Imagine a system of linear equations: A · s = b. If you know A and b, you can solve for the secret vector s using standard linear algebra. Now add small random noise to each equation: A · s + e ≈ b. Recovering s from the noisy equations is computationally hard — even with a quantum computer. The noise is the security. Shor's algorithm exploits the periodicity of modular arithmetic in RSA and ECC — but noisy linear algebra has no such exploitable structure. There is no known quantum algorithm that solves LWE significantly faster than the best classical algorithm.
Classical Problem (Solvable)
3s₁ + 2s₂ + s₃ = 14
s₁ + 4s₂ + 2s₃ = 19
2s₁ + s₂ + 3s₃ = 15
Gaussian elimination → s = [1, 2, 3]. Trivially solvable. Not secure.
LWE Problem (Quantum-Hard)
3s₁ + 2s₂ + s₃ + 1 = 15
s₁ + 4s₂ + 2s₃ + -2 = 17
2s₁ + s₂ + 3s₃ + 1 = 16
Small noise e added. Recovering s is computationally hard. Even quantum Fourier transforms can't exploit the structure because there is no periodic structure to exploit.
❓ Why Not Just Remove the Noise? Because the noise is random and unknown to the attacker. Without knowing which noise value was added to each equation, there is no way to cleanly separate signal from error. This is the LWE hardness assumption — and it has resisted 30 years of cryptanalysis.
✅ How ML-KEM Uses LWE ML-KEM (CRYSTALS-Kyber) encodes its public key as a noisy LWE instance. Encapsulation creates ciphertext that can only be "decoded" correctly by the private key holder who knows the secret s. Anyone else only sees noise.

As noise increases, the equations become inconsistent: any guess introduces accumulated errors across dimensions. ML-KEM operates at noise levels equivalent to the right end of this scale, applied across vectors of 256+ elements.

Key Exchange (KEX) and Key Encapsulation (KEM) both produce a shared secret. They do it differently.

Key Exchange — ECDH Pattern
1. Alice generates key pair (pkₐ, skₐ)
2. Bob generates key pair (pk_b, sk_b)
3. Alice computes: K = sk_a × pk_b
4. Bob computes: K = sk_b × pk_a
5. Both arrive at same K

Both sides contribute equally to K. Requires both to be online simultaneously. Broken by Shor's.
Key Encapsulation — ML-KEM Pattern
1. Alice generates key pair (pk, sk)
2. Alice publishes pk
3. Bob runs: ct, K = Encap(pk)
4. Bob sends ciphertext ct to Alice
5. Alice runs: K = Decap(sk, ct)
Bob generates and wraps the secret. Alice unwraps it. Async-friendly. Quantum-safe.
⚠ Why This Design Difference Matters In a KEM, the shared secret is chosen by the encapsulator (Bob) and transmitted wrapped inside the ciphertext. Alice's private key is the only way to unwrap it. This design is async-friendly and maps cleanly onto TLS 1.3's key_share extension. ECDH required real-time mutual computation; ML-KEM does not.

NIST standardized three security levels for ML-KEM, calibrated against the cost of the best known classical and quantum attacks.

Variant NIST Level Classical Equiv Public Key Use Case

🎯 Which Level to Choose? NIST recommends ML-KEM-768 for most applications. It provides security equivalent to AES-192 — well above the 128-bit threshold at which security is generally considered unbreakable for the foreseeable future. ML-KEM-1024 carries a size and performance overhead only justified for applications requiring data confidentiality beyond 30+ years.

# ML-KEM Key Exchange — NIST FIPS 203
import oqs
# Step 1: Alice generates a keypair
kem = oqs.KeyEncapsulation('ML-KEM-768')
public_key = kem.generate_keypair()
# Step 2: Bob encapsulates a secret using Alice's public key
ciphertext, shared_secret_bob = kem.encap_secret(public_key)
# Step 3: Alice decapsulates using her private key
shared_secret_alice = kem.decap_secret(ciphertext)
# Step 4: Verify both parties have the same secret
print(f"Secrets match: {shared_secret_alice == shared_secret_bob}")

ML-KEM vs RSA vs ECDH — The Tradeoff

Run the benchmark simulation across all five algorithms. Observe where ML-KEM wins, where it costs more, and what the practical implication is for deployment.

lab2_benchmark.py — 100 iterations per algorithm
# Benchmark ML-KEM-512/768/1024 vs RSA-2048 vs ECDH P-256
import oqs, time, statistics
algorithms = ['ML-KEM-512', 'ML-KEM-768', 'ML-KEM-1024']
for alg in algorithms:
    times = []
    for _ in range(100):
        kem = oqs.KeyEncapsulation(alg)
        t0 = time.perf_counter()
        pk = kem.generate_keypair()
        ct, ss_b = kem.encap_secret(pk)
        ss_a = kem.decap_secret(ct)
        times.append(time.perf_counter() - t0)
    print(f"{alg}: avg={statistics.mean(times)*1000:.3f}ms")

Reflection · Closing the Loop

Consolidate What You Built

Three questions that test whether you understand not just what ML-KEM does, but why it was designed the way it was.

1. In the bit flip challenge, ML-KEM returned a structurally valid but incorrect shared secret — no exception was thrown. What does this mean for protocol design?
A
Nothing — the incorrect secret will be immediately obvious because the session will fail to decrypt any payload.
B
ML-KEM is broken and should not be deployed.
C
ML-KEM is an IND-CCA2 secure primitive — it guarantees confidentiality, not authentication. Silent failure means any system using ML-KEM must add a message authentication layer (e.g., HMAC or an authenticated cipher like AES-GCM) to detect tampering. The KEM is working as designed; authentication is a separate concern.
D
The bit flip would be detectable if TLS were used, making this a non-issue in practice.
2. From the benchmark, RSA-2048 key generation is roughly 200× slower than ML-KEM-768 key generation. Yet many engineers argue RSA's smaller key size is an advantage. When would this tradeoff actually favor RSA?
A
Never — ML-KEM is strictly better in every dimension.
B
In bandwidth-constrained environments (legacy IoT with tiny packet sizes) or embedded systems where the 1184-byte ML-KEM-768 public key causes significant overhead, the smaller RSA key (294 bytes) might appear attractive. However, this is a false choice — ML-KEM-512 (800 bytes) addresses most constrained scenarios while maintaining quantum safety. RSA's slow keygen also makes it poorly suited for ephemeral key generation.
C
When the application does not need quantum resistance, making RSA-2048 the permanent choice.
D
RSA's key size advantage is only relevant for digital signatures, not key exchange.
3. Why might an organization choose to run RSA and ML-KEM simultaneously during a migration period rather than switching entirely to ML-KEM?
A
Because ML-KEM is experimental and not yet trusted for production use.
B
To save money — ML-KEM licenses are more expensive than RSA.
C
Hybrid deployment provides defense-in-depth: if a future vulnerability is discovered in ML-KEM's lattice assumptions, the RSA component still protects classical adversaries (and vice versa). It also allows gradual migration without breaking compatibility with legacy clients. NIST and major browser vendors explicitly recommend hybrid X25519+ML-KEM during the transition period for exactly this reason.
D
Because ML-KEM cannot yet interoperate with TLS without RSA as a fallback.

Hybrid Key Exchange — How It Works in TLS 1.3

The hybrid approach combines an ECDH (or X25519) key share with an ML-KEM key share. The final session key is derived from both. An attacker must break both to compromise the session.

✅ Why Hybrid Is the Recommended Migration Path Google Chrome, Firefox, and Cloudflare all deployed hybrid X25519+ML-KEM in 2024. The combined key is no weaker than its strongest component. If ML-KEM is someday broken classically, X25519 still protects you. If a classical adversary somehow cracks X25519, ML-KEM still protects you against quantum decryption of archived traffic. You only lose if both are broken simultaneously.
