SECTION 1 OF 7
Concept · Signature Forgery + ML-DSA Overview

When Signatures Can Be Forged, Nothing Is Trustworthy

The intuition first. Then a comparison table. Then a micro-check before you implement.

💡 The Core Idea — Signature Forgery A digital signature works because only the private key holder can produce a valid signature — anyone with the public key can verify it, but not create it. Shor's algorithm breaks this by computing the private key directly from the public key. Once an attacker has the private key, they can produce signatures that are mathematically indistinguishable from legitimate ones — on any document, past or future. This is not a decryption attack: it is full identity theft at the cryptographic level.
Encryption Compromise

An adversary who breaks RSA encryption can read past recorded ciphertext. The data is exposed retroactively. Future communications using new keys remain safe after rekeying.

Impact: Past data exposed
Signature Forgery

An adversary who recovers an ECDSA private key can forge any future signature as that entity — software updates, contracts, certificates, code. Rekeying old signed documents doesn't help: old valid signatures remain trusted.

Impact: Past AND future identity compromised
🏥 The Hospital Attack in Detail The adversary harvested signed software updates from the vendor in 2024. These updates contained the vendor's ECDSA public key in the signature. In 2032, a quantum computer recovers the vendor's private key from those public keys. The adversary now signs a malicious binary as the vendor. Every hospital running that vendor's update infrastructure will install it — and every signature check will pass. There is no cryptographic signal that anything is wrong.

📐 The Size Cost An ECDSA signature is 72 bytes. An ML-DSA-65 signature is 3309 bytes — roughly 46× larger. This is the primary deployment tradeoff for signatures. For large files this is negligible. For small messages (firmware headers, embedded certificates, IoT telemetry packets) it requires careful planning.

ML-DSA (CRYSTALS-Dilithium) is a lattice-based signature scheme built on Module Learning With Errors. Signing produces a vector that proves knowledge of the private key without revealing it.

ml_dsa_overview.py — conceptual walkthrough
# Key Generation
signer = oqs.Signature('ML-DSA-65')
public_key = signer.generate_keypair() # pk: 1952 B, sk: 4032 B
# Signing — private key produces a lattice "witness"
message = b"DEPLOY firmware v4.3.1 to ward-7 cardiac monitors"
signature = signer.sign(message) # sig: 3309 B
# Verification — public key checks the witness against the message
is_valid = signer.verify(message, signature, public_key)
print(is_valid) → True
# Tampered message — even one byte change invalidates the signature
tampered = b"DEPLOY firmware v4.3.2 to ward-7 cardiac monitors"
is_valid = signer.verify(tampered, signature, public_key)
print(is_valid) → False # Cryptographic binding holds
✅ What Makes This Quantum-Safe Verifying a signature requires finding a short vector in a lattice — which is computationally hard even with quantum computers. Recovering the private key from the public key requires solving the Module Learning With Errors problem, which has no known quantum speedup. Unlike ECDSA, there is no algebraic period structure for Shor's Fourier analysis to exploit.

The ECDSA forgery attack follows a specific timeline. Understanding it makes the urgency concrete.

2024
Vendor publishes signed updates. Attacker collects public keys.
2028–32
Quantum computer arrives. Attacker runs Shor's on ECDSA public keys.
2032+
Private key recovered. Attacker forges updates indefinitely. No detection possible.
⚠ Unlike Encryption, You Cannot Retroactively Rekey With encryption, rekeying after a compromise protects future data. With signatures, once the private key is recovered from historical public keys, the attacker can sign new documents with that key at any time — forever. The CA certificate revocation system helps, but only if you know the key is compromised. A quantum forgery attack is silent until the malicious payload executes.

Exercise C · File Signing — Signature Overhead Ratio

How Much Does the Signature Add?

Simulate signing files of different sizes. Measure the ratio of signature size to file size. Identify where the overhead starts to matter — and where it is negligible.

lab3_filesign.py — Exercise C
# Sign files of varying sizes and measure overhead ratio
import oqs, os, hashlib
signer = oqs.Signature('ML-DSA-65')
pk = signer.generate_keypair()
file_sizes = [512, 4096, 65536, 1048576, 10485760] # 512B → 10MB
for size in file_sizes:
    data = os.urandom(size)
    digest = hashlib.sha3_256(data).digest() # hash then sign
    sig = signer.sign(digest)
    ratio = len(sig) / size * 100
    print(f"File: {size/1024:.1f} KB | Sig: {len(sig)} B | Overhead: {ratio:.2f}%")

🏥 Practical Implication — The Hospital Context

A cardiac monitor firmware update is typically 2–8 MB. The ML-DSA-65 signature of 3309 bytes represents less than 0.02% of a 2 MB file. For this use case, the signature overhead is operationally irrelevant. The performance cost of signing is similarly negligible for a file signed once and deployed thousands of times. There is no practical justification for keeping ECDSA in this deployment context.

Reflection · Closing the Loop

Signatures, Trust, and the Quantum Transition

Three questions that connect what you built to how signature infrastructure actually works at scale.

1. In the forgery grid, you tried 16 combinations of message and signature tampering. None of them produced a valid signature. What does this tell you about the effort required to forge an ML-DSA signature by trial and error?
A
If you tried enough combinations, you would eventually find one that works — it just requires many attempts.
B
Trial and error is viable with a fast computer — brute force over the 3309-byte signature space would find a forgery within days.
C
A valid signature requires knowledge of the private key. Without it, no modification to the message or signature produces a valid pair — there is no "close enough." The search space is 2^(3309×8) possible signatures, which is astronomically larger than any computable search. The forgery grid proves this empirically: even systematic exploration finds nothing.
D
ML-DSA can be brute forced more easily than RSA because its signatures are larger.
2. The hospital scenario describes a forged software update that passes signature verification. What additional defense-in-depth measures could a hospital deploy that would catch this attack even if signature verification passes?
A
Nothing — if the signature verifies with the correct public key, the update must be legitimate. There is no defense if the signing key is compromised.
B
Defense-in-depth measures include: code signing with multiple independent CAs (attacker must compromise all), binary transparency logs (append-only ledgers of all signed releases that can be audited), runtime behavior monitoring (malicious update behaves differently from a legitimate one), network segmentation (limits damage even after installation), and hash pinning (comparing against a known-good hash from a separate channel). Cryptographic signature verification is necessary but not sufficient for supply chain security.
C
Using a hardware security module (HSM) for key storage would prevent the attack because private keys in HSMs cannot be exported.
D
Requiring TLS for update delivery would prevent the forged update from being installed.
3. An ML-DSA-65 public key is 1952 bytes. An ECDSA P-256 public key is 64 bytes. A root CA certificate is distributed to every device in a PKI ecosystem — potentially billions of endpoints. Which of the following best describes the practical implication of this size difference?
A
The size difference makes ML-DSA impractical for PKI — root CA certificates are too large to distribute to IoT devices.
B
Root CA certificates are distributed once and stored locally — they are not retransmitted during normal operations. A 1952-byte public key adds about 1.9 KB to a root CA certificate, which is stored in firmware and never transmitted in TLS handshakes. The operational impact is minimal for most devices. Constrained IoT with extremely limited firmware storage (under 32 KB total) may need careful planning, but this is a solvable engineering problem, not a fundamental barrier.
C
The size difference only matters during key generation — once distributed, public keys do not consume any additional bandwidth.
D
ML-DSA public keys can be compressed to the same size as ECDSA keys using standard zlib compression.
